# Use Cases: bio4j-api-gateway

## UC-01. Авторизованный запрос в целевой микросервис

- **Цель:** пропустить запрос в downstream сервис только для валидного пользователя.
- **Акторы:** клиент (frontend/сервис), API Gateway, Redis, БД пользователей, целевой микросервис.
- **Предусловия:**
  - клиент отправляет JWT в `Authorization: Bearer <token>`;
  - в JWT есть claim `sberpdi`;
  - пользователь с таким `login` существует в БД и активен.
- **Основной поток:**
  1. Клиент отправляет REST запрос в gateway.
  2. Gateway валидирует JWT (подпись, срок действия).
  3. Gateway извлекает `sberpdi` из JWT.
  4. Gateway загружает профиль пользователя из БД (`id`, `login`, `roles`, `grants`).
  5. Gateway определяет `sessionId` (`sid` -> `jti` -> входящий `X-Session-Id` -> UUID).
  6. Gateway сохраняет профиль в Redis по ключу `session:user-profile:{sessionId}`.
  7. Gateway добавляет заголовок `X-Session-Id: {sessionId}` в downstream запрос.
  8. Gateway маршрутизирует запрос в целевой сервис по правилам роутера.
- **Постусловия:**
  - целевой сервис получает `X-Session-Id`;
  - профиль пользователя доступен в Redis по `sessionId`.

## UC-02. JWT отсутствует

- **Цель:** отклонить неавторизованный запрос.
- **Предусловия:** отсутствует заголовок `Authorization` или формат Bearer некорректен.
- **Поток:**
  1. Клиент отправляет запрос без корректного Bearer токена.
  2. Gateway не проходит аутентификацию.
  3. Gateway возвращает `401` JSON.
- **Результат:** запрос не маршрутизируется в downstream.

## UC-03. JWT невалиден или истек

- **Цель:** не пропускать запросы с недействительным токеном.
- **Предусловия:** JWT имеет неверную подпись или `exp` в прошлом.
- **Поток:**
  1. Gateway выполняет валидацию JWT.
  2. Валидация завершается ошибкой.
  3. Gateway возвращает `401` JSON (`JWT token is invalid or expired`).
- **Результат:** запрос блокируется.

## UC-04. В JWT нет claim `sberpdi`

- **Цель:** гарантировать наличие идентификатора пользователя.
- **Предусловия:** JWT валиден, но claim `sberpdi` отсутствует/пустой.
- **Поток:**
  1. Gateway валидирует JWT.
  2. Gateway пытается извлечь `sberpdi`.
  3. Claim отсутствует.
  4. Gateway возвращает `401` JSON (`JWT claim 'sberpdi' is missing`).
- **Результат:** запрос блокируется.

## UC-05. Пользователь не найден в БД или неактивен

- **Цель:** пропускать только зарегистрированных и активных пользователей.
- **Предусловия:** JWT валиден и содержит `sberpdi`, но запись в БД не найдена или `active=false`.
- **Поток:**
  1. Gateway получает `sberpdi`.
  2. Gateway ищет пользователя в БД (`findByLoginAndActiveTrue`).
  3. Пользователь не найден.
  4. Gateway возвращает `401` JSON (`User login from JWT not found in database`).
- **Результат:** запрос блокируется.

## UC-06. Получение профиля на целевом сервисе

- **Цель:** целевой сервис поднимает профиль автоматически из Redis.
- **Акторы:** целевой сервис, Redis.
- **Предусловия:**
  - в запросе есть `X-Session-Id`;
  - в Redis существует ключ `session:user-profile:{sessionId}`.
- **Поток:**
  1. Целевой сервис читает `X-Session-Id`.
  2. Формирует Redis-ключ `session:user-profile:{sessionId}`.
  3. Загружает профиль из Redis.
  4. Использует `roles` и `grants` для авторизации бизнес-операций.
- **Альтернативный поток:**
  - если ключ отсутствует (TTL истек), сервис может вернуть `401/403` или запросить ре-авторизацию (зависит от политики сервиса).

## Формат ответа об ошибке авторизации

Во всех негативных сценариях gateway возвращает:

- HTTP статус: `401 Unauthorized`
- JSON:
  - `timestamp`
  - `status`
  - `error`
  - `message`
  - `path`
